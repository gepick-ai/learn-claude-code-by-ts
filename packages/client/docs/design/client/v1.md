# @gepick/client：项目工程技术选型（定稿）

本文只谈一件事：**不算业务代码本身，一个现代前端子包在工程上应该长什么样**——用什么工具写、怎么检查质量、怎么放进大仓库里一起维护。  
写页面、调接口、状态怎么存，见同目录 **[与 gepick-app 配套对接说明](./与-gepick-app-配套对接说明.md)**，**不要**和本文混在一起。

---

## 这份文档是给谁看的

- 要**新建 / 维护** `packages/client` 的同学。  
- 做 **Code Review** 时对照：工程项是否齐、风格是否一致。  
- 与具体产品需求**解耦**：换需求，下面的「搭房子」方式仍然成立。

---

## 我们到底要达成什么（四句话）

1. **写得快**：本地改代码能立即在浏览器里看到效果（热更新），不用每次全量打包再等半天。  
2. **少踩坑**：TypeScript 帮你在写的时候抓类型问题；ESLint 抓常见逻辑/写法问题。  
3. **长得一样**：Prettier 统一缩进、换行，减少「只改格式」的无意义 diff。  
4. **能上线**：一条命令打出**固定目录**（一般是 `dist/`）的静态文件，交给网关或 Node 去托管即可。

## 技术栈（工程向，已定）

| 用途 | 选型 | 白话 |
|------|------|------|
| 语言 | **TypeScript** | 全用 TS 写，和根仓库其它包习惯一致；具体 `strict` 等开关跟团队 tsconfig。 |
| 本地开发 + 打包 | **Vite** | 开发时启动快、改完即刷新；发布时打正式包，默认输出到 **`dist/`**。 |
| 界面 | **React** | 用函数组件 + Hooks，业界默认，资料多。 |
| 样式 | **Tailwind CSS v4** + **@tailwindcss/vite** | 用工具类拼 UI，少写大段自定义 CSS；v4 用官方 Vite 插件接进项目。 |
| 让 React 跑在 Vite 里 | **@vitejs/plugin-react** | 需要更快编译时可再考虑换成 **SWC** 版插件。 |
| 条件 class | **clsx**、**tailwind-merge** | 动态拼 `className` 时不乱、不冲突。 |
| 路径 | **`@/` 之类别名** | `import` 不用 `../../../`，在 `vite.config` 和 `tsconfig` 里**成对**配好。 |

**刻意不在这里定死的**（交给业务或单独一页）：用不用 Zustand、上不上 React Router、请求用不用 TanStack Query 等——和「盖楼工具」无关，随产品变。  

**以后再加也常见**：单测（Vitest + Testing Library）、Storybook、路由等，有需求再补依赖和脚本即可。

**样式加分项（可选）**：`@tailwindcss/typography`（大段排版）、设计 token（CSS 变量）等。

## 代码质量：ESLint + Prettier（首版就带上）

- **ESLint**：管「有没有写错、有没有反模式」——建议接 **TypeScript**（`typescript-eslint`）和 **React Hooks** 规则（`eslint-plugin-react-hooks`），配置文件用社区常用的 **flat config**（`eslint.config.*`），以后好改。  
- **Prettier**：管「长什么样」——缩进、引号、换行统一。  
- **两者别打架**：加 **eslint-config-prettier**，关掉和 Prettier 重复的格式规则，避免保存时 ESLint 和 Prettier 各改各的。  

**放哪**：配置放在 **`@gepick/client` 包自己目录里**（例如 `eslint.config.js`、`.prettierrc` 或 `package.json` 里的 `prettier` 字段），在 `package.json` 里写 **`lint`**、**`format`** 脚本；CI 里可对 `src` 跑 `eslint`、对仓库跑 `prettier --check`。  

**扫谁**：主要扫本包 **`src` 下的 `.ts` / `.tsx`**；**不要**扫 `dist`、锁文件、生成物。

## 仓库里怎么摆：Bun + Monorepo + 版本表（catalog）

本仓库用 **Bun** 装依赖；根目录的 **`package.json` 里有一个 `workspaces`**，把多个子目录当成**多个包**一起管（例如 `packages/client`、`packages/app`、`packages/sdk`、根目录 `tutorial/` 等）。

**为什么要提 catalog？**  
多个包常常依赖**同一批**库（比如大家都用同一个 `zod` 大版本）。Version 写在根上的 **`workspaces.catalog` 里只写一次**，各子包在 `dependencies` 里写 **`"包名": "catalog:"`**，表示「版本跟总表走」。以后**升版本只改总表**，避免每个包各写一版、对不齐。

**和写业务的关系**：这是**仓库和依赖怎么管**，不等于「怎么调你的 HTTP 接口」；对接方式仍在《配套对接说明》里写。

**日常**：在**仓库根**执行一次 **`bun install`** 即可。根 `package.json` 里可以写**快捷脚本**，用 **`bun run -F 包名 脚本名`** 只跑某个子包的 `dev` / `build` 等（包名以各子包 `package.json` 的 `name` 为准，例如 `@gepick/client`）；在 zsh 下对含 `@` 的包名建议写成 **`bun run -F '@gepick/client' dev`**。

## 和「只谈工程」相关的运行细节（不涉及具体业务 URL）

- **环境变量（浏览器能看到的）**：键名以 **`VITE_` 开头**，在代码里用 `import.meta.env` 读。真正的密钥、内网地址别提交进 git，用本机 **`.env.local`** 等忽略文件解决。  
- **开发时代理**：Vite 里配 **`server.proxy`**，把某些路径转到你的后端。具体转给谁、转哪几条路径，**按对接文档**来；工程选型只要求：「项目里**能配** Vite 代理」即可。  
- **发布形态**：一般是 **单页应用（SPA）**——`build` 出一堆静态文件，服务器上对「找不到文件」的请求**回到 `index.html`**，由前端路由接管。谁来托管（Hono、Nginx、对象存储）是部署细节，不在这里定死。
