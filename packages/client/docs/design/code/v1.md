# Code 业务区 v1

**状态**：v1 需求说明（最小闭环）  
**关联**：[会话界面 v5](../session/v5.md)（四区布局与 `slot` 入口）

---

## 1. 设计目标

v1 目标：**在现有 agent 底座上完成 code 业务的最小闭环**。

`code` 业务定位为：用户输入驱动 Agent 写代码，替代人类工程师的基础开发过程，并在 `session` 的业务 `slot` 中直接运行结果。

v1 闭环定义：

1. 用户在 Chat 输入开发需求（例如“生成一个贪吃蛇游戏”）；
2. Agent 输出可执行代码（v1 为最简单 HTML）；
3. 前端提取 HTML 并在 `slot` 内渲染运行；
4. 用户可直接交互，并继续输入迭代需求。

---

## 2. 技术选型

围绕“最小闭环可用”原则，v1 技术选型如下：

- 消息与增量链路：复用现有 `session + SSE`，不新增后端协议。
- 可执行代码来源：解析 Agent 文本输出中的 HTML。
- 渲染容器：使用 `iframe + srcdoc` 运行 HTML，隔离宿主页面。
- 状态存储：前端内存态（按 `sessionId` 维护）。
- 页面挂载：`code` 业务区必须嵌入 `session` 的 `slot`，不单独新增页面入口。

---

## 3. 架构与数据流

### 3.1 角色边界

- `session`：负责会话生命周期、消息发送/拉取、SSE 合并。
- `code`：负责从 assistant 输出提取 HTML，构建业务区视图模型，并完成渲染。

### 3.2 数据流

1. Chat 发送需求，消息进入当前 `session`。
2. SSE 持续推送 assistant `text` 增量。
3. `code` 模块在消息更新后执行 HTML 提取。
4. 提取成功则更新 `code` 内存态并刷新业务区 iframe。
5. 提取失败则保持聊天链路正常，仅更新业务区状态为 `error` 或维持旧值。

---

## 4. v1 实现范围

### 4.1 输入约定

- 数据来源：assistant 的 `text` 内容。
- 解析目标：第一段合法完整 HTML。
- 兼容格式：
  - 纯 HTML 文本；
  - Markdown `html` 代码块中的 HTML 内容。

说明：若 SSE 增量尚不完整（半包），暂不更新业务区，等待后续补齐再解析。

### 4.2 渲染与交互

- `code-panel` 渲染在 `session-slot-panel` 内部。
- 使用 iframe 承载 HTML，支持基础键盘/鼠标交互。
- 提供“重新加载”能力，用于重置当前运行态。

### 4.3 状态模型（内存态）

按 `sessionId` 维护：

- `generatedHtmlBySession`
- `codePanelStatusBySession: empty | ready | error`
- `codePanelErrorBySession`（可选）

页面刷新后状态丢失，符合 v1 范围。

### 4.4 目录建议

```text
packages/client/src/session/
  session-page.tsx
  session-store.ts
  slot/
    session-slot-panel.tsx                # code 业务挂载入口
  code/
    code-panel.tsx                        # empty/ready/error 渲染
    extract-html-from-message.ts          # 提取 HTML
    build-code-view-model.ts              # 消息 -> code 视图模型
```

---

## 5. 非目标

v1 不包含以下能力：

- 多文件工程与文件树管理；
- 后端 artifact 持久化、版本管理与回放；
- 导出/下载/部署；
- 在线编辑器与高级调试能力；
- 新增后端 API。

---

## 6. 验收标准（DoD）

- 用户发送“生成一个贪吃蛇游戏”后，`slot` 内可渲染并可玩。
- 业务区运行在 iframe 中，不污染 Chat 区样式与行为。
- 切换不同会话时，业务区按 `sessionId` 回显内存态。
- 刷新页面后结果丢失（符合 v1 设计预期）。
- 不影响现有会话发送、历史加载、SSE 增量展示能力。
