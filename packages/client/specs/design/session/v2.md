# 会话界面 v2

**状态**：v2 需求说明（助手消息 Markdown 呈现 + 技术选型）  
**关联**：[会话界面 v1](./v1.md)（布局与交互，v2 不改变）；[与 gepick-app 配套对接说明](../与-gepick-app-配套对接说明.md)（数据流、API、Zustand、SSE）

---

## 1. 目的与范围

在 **v1 已约定的左右分栏、会话列表、对话流与发送区** 不变的前提下，v2 专门约定：

- **助手侧**随 SSE **流式增长**的正文与推理内容，应按 **Markdown** 渲染为富文本（标题、列表、代码块、链接等），**不得**长期以原始 `**`、`#`、围栏等源码形式展示。
- **用户侧**消息在 v2 中仍按 **纯文本** 展示（与输入框一致，见 §5）。

Store、`session.part.delta` / `session.part.updated` 的合并策略仍以《配套对接》与 `applySseToMessages` 为准；v2 **不要求**为 Markdown 单独改服务端协议。

---

## 2. 技术选型

### 2.1 选型目标

在 **React + Zustand + SSE 增量文本** 前提下，将助手消息从「源码串」变为「安全、可样式化的富文本」，并控制依赖体积与维护成本。

### 2.2 候选方案对比

| 方案 | 要点 | 结论 |
|------|------|------|
| **A. `react-markdown` + `remark` / `rehype`** | 将 Markdown 解析为 **React 元素**；默认对原始 HTML **转义**；`remark-gfm` 可扩展 GFM；`components` 映射便于与 Tailwind 统一。生态见 [remarkjs/react-markdown](https://github.com/remarkjs/react-markdown)。 | **采用（v2 主路径）** |
| **B. `streaming-markdown` 等** | 按 chunk 增量解析，对未闭合语法更友好；常见为命令式写 DOM，与「`part.text` 驱动 React 树」集成成本较高。见 [thetarnav/streaming-markdown](https://github.com/thetarnav/streaming-markdown)。 | **备选**（若对流式中间态极度敏感再评估） |
| **C. `markdown-it` / `marked` → HTML 字符串** | 易落到 `dangerouslySetInnerHTML`，安全与组件粒度成本高。 | **不采用** |
| **D. `markdown-to-jsx`** | 轻量但需单独评审 HTML 与安全策略；与 A 重叠。 | **不优先** |

### 2.3 定稿结论

- **渲染栈**：`react-markdown` + **`remark-gfm`**（表格、任务列表、删除线、自动链接等 GFM 能力）+ **`rehype-sanitize`**（净化输出，防御 XSS）。
- **禁止**：在助手消息渲染路径上启用 **`rehype-raw`** 或等价「原样输出 HTML」能力，除非另行通过安全评审并单独成文。
- **兜底**：保留对 Markdown 内嵌 HTML 的默认转义行为；与 `rehype-sanitize` 组合使用时，以官方推荐顺序在 `rehypePlugins` 中配置（先变换再净化，具体以实现时依赖文档为准）。
- **版本与安装**：依赖加在 **`packages/client`**；若 monorepo 使用 catalog，可将 `react-markdown`、`remark-gfm`、`rehype-sanitize` 纳入根目录 `package.json` 的 `catalog` 并在子包中 `catalog:` 引用，否则在子包 `dependencies` 中直接指定兼容版本。
- **排版样式**：首版使用 `react-markdown` 的 **`components` 映射 + Tailwind**，与现有气泡内 `text-sm`、`slate-*` 等风格对齐；**不**强制引入 `@tailwindcss/typography`（与《项目工程技术选型》中「可选 typography」一致，列为加分项即可）。
- **代码高亮**：v2 **不**强制语法高亮库；围栏代码块以 `<pre>` / `<code>` + 横向滚动与等宽字体为主。彩色高亮（如 `react-syntax-highlighter`）列为后续迭代。
- **流式策略**：前端在 **`part.text` 每次更新时对当前完整字符串全量重解析**；不要求首版实现「流式 Markdown 补全」或节流（若性能反馈再议 `memo` / `requestAnimationFrame` 等）。

---

## 3. 功能目标

- **`text` part（助手）**：在 SSE 流式过程中持续以 Markdown 富文本展示。
- **`reasoning` part（助手）**：**同样**按 Markdown 渲染；视觉上与正文区分（沿用或微调 v1 实现中的左边框、斜体等语义，避免与正文完全雷同即可）。
- **Tool 等其它 part**：仍按《配套对接》与现有类型展示；v2 不重新定义 Tool UI。

---

## 4. Markdown 能力范围（v2）

在 `remark-gfm` 启用前提下，产品侧默认承诺至少包括：

- 段落、换行；**粗体**、*斜体*；行内 `` `code` ``。
- 有序 / 无序列表；引用块（若解析器默认支持）。
- 链接；图片（`alt`、加载失败表现以实现为准）。
- 围栏代码块（语言标签可选）；**不**承诺 v2 内建语法着色。
- GFM：**表格**、**任务列表**、**删除线**、自动链接等（随 `remark-gfm` 能力）。

**不纳入 v2**：数学公式（KaTeX 等）、Mermaid 图表、自定义 HTML 组件嵌入；需要时另开 v3+ 说明。

---

## 5. 用户消息

- v2 **仅**将助手侧 `text` / `reasoning` 走 Markdown 管线。
- **用户**消息保持 **纯文本**（如 `whitespace-pre-wrap` 保留换行），**不**走 `react-markdown`，以降低误解析与用户粘贴内容带来的安全面。

可选扩展：若产品后续要求用户气泡也支持 Markdown，在修订记录中单独立项并补充安全与样式约定。

---

## 6. 安全与链接

- **不信任**助手 Markdown 中的 HTML：采用 §2.3 的 **`rehype-sanitize`** + 禁用 raw HTML 管线。
- **外链**：`a` 元素应 **`target="_blank"`** 且 **`rel="noopener noreferrer"`**（通过 `components.a` 统一实现）。
- 助手内容视为 **不可信输入**；禁止执行 Markdown 内脚本或内联事件处理器（由 sanitize 与默认转义保障）。

---

## 7. 流式体验与已知限制

- 流式过程中字符串可能处于 **未闭合** 语法（例如半个 `**`、未闭合的 \`\`\`），解析结果会在后续 token 到达后变化；**允许**短暂视觉中间态，**不**要求首版实现「补全括号再渲染」类算法。
- **围栏代码块**：在结束围栏到达前，内容可能暂时落在普通段落中；属常见权衡，可在后续版本引入更强流式解析优化。

---

## 8. 视觉与可访问性

- 列表、标题层级在气泡内应有 **清晰缩进与字号阶梯**，避免与正文糊成一块。
- **代码块**：`pre` 区域 **`overflow-x: auto`**，小字号等宽字体；长行不换行或按产品统一策略。
- 图片需有合理 **`alt`**（若模型未提供，可实现层可留空但结构保留）。
- 与 v1 一致的对比度与可读性目标；暗色主题细稿仍可由全局设计迭代补充。

## 9. 设计代码目录（在 v1 基础上增量）

v1 已约定 [`src/session/chat/`](../../src/session/chat/) 承载消息区与发送区。v2 **新增** Markdown 呈现组件，建议：

```text
packages/client/src/
  session/
    chat/
      MessageTranscript.tsx    # 按 role / part 分流：助手 → MarkdownMessageBody
      MarkdownMessageBody.tsx  # 封装 react-markdown + plugins + components 映射（新建）
      ...
```

职责说明：

| 文件 | v2 职责 |
|------|---------|
| `MarkdownMessageBody.tsx` | 接收 `content: string` 与可选变体（如 `variant: "reasoning" \| "default"`），内部组装 `ReactMarkdown`、`remarkGfm`、`rehypeSanitize`、`components`。 |
| `MessageTranscript.tsx` | 助手 `text` / `reasoning` 使用 `MarkdownMessageBody`；用户 `text` 仍为纯文本节点；Tool 等不变。 |

**不**因 v2 新建与 `history`、`chat` 同级的第三会话子域目录。
