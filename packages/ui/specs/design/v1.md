# @gepick/ui 设计说明（v1）

## 定位

`@gepick/ui` 是 gepick 内多产品共用的 **UI 组件与设计体系**（组件、统一样式/主题、类型），只做 **界面复用与体验一致**；不承载业务后端、鉴权、路由与数据层。目标是把多端重复造界面、品牌与 token 分散、无障碍难统一等问题收束到**一个可 import 的包**里。与 **Ant Design**、**Element Plus** 相比，**角色相同**（都是可复用的组件包），差异是前两者为**通用、大而全**生态，`@gepick/ui` 只在 **gepick 域内渐进建设**，不对标组件数量与 API 面。**默认**不承担 BFF、鉴权、Store、产品级路由编排；例外须在文档单独约定；也不保证与 Ant/Element 的 API 可一一迁移。

## 开发流程

### 一、开发时唯一要记的主线

**改库里的代码 → 在 Storybook 里看对不对 → typecheck 过 → 再去业务里接。**

- **主战场**：Storybook（在仓库根执行 `bun -F @gepick/ui dev`，或在 `packages/ui` 里执行 `bun storybook`；默认端口为 **6006**）。  
- **验收画面**：各组件的 `*.stories.*`，**状态、尺寸、主题**能覆盖尽量覆盖。  
- **业务项目**（如 `@gepick/client`）：用于验证**真路由、真数据、真布局**下也成立，是**第二步**，不是第一步。  

**仓库布局**：`ui/.storybook` 与 `ui/src` 同包即可；将 Storybook 拆成独立子包也可以，只影响组织方式。  

> **Tip**  
> 基于 **Storybook 驱动** = 主战场在 Storybook；**同一轮迭代的默认循环**是「改组件 → 改 Story 验收」，进业务是**后一步**集成。Storybook 只服务**开发/文档/验收**（可选部署静态站作设计台）；**不是**业务运行时依赖，用户侧最终 `import` 的仍是**包里的组件与样式**。

### 二、一天里常见的具体动作

| 你想做的事 | 怎么做 |
|------------|--------|
| **起环境** | 在仓库根：`bun -F @gepick/ui dev`（或进入 `packages/ui` 后执行 `bun storybook`），浏览器打开默认端口（**6006**）。 |
| **改现有组件** | 改 `src/components/...`，并同步改同目录或旁边的 `*.stories.*`；在 Storybook 左侧进入对应 Story，保存后刷新查看。 |
| **加新组件** | 在 `src/components` 新建组件，并新增 `xxx.stories.tsx`；Story 中先列出**默认、若干 variant、disabled** 等状态。 |
| **动样式 / token** | 改 `src/styles/globals.css` 或 `src/theme`；`preview` 里已包 `ThemeProvider` 和全局样式，在 Storybook 中查看是否影响各 Story。 |
| **加可复用 hook** | 写在 `src/hooks` 或 `src/theme`；需要展示时，可写**仅给 Story 用**的小例子，或写在某个组件的 Story 里。 |
| **提交前** | 在 `packages/ui` 执行 `bun typecheck`；若需全仓校验，在仓库根执行 `bun typecheck:packages`。 |

### 三、加一个新组件的推荐顺序（实操）

1. 在 `src/components` 写组件（先满足**一种**主状态即可）。  
2. 编写 `组件名.stories.tsx`：`title` 建议为 `Components/xxx`，先建 **Default**，再按需加 **Primary / Disabled / …**。  
3. 在 Storybook 中点选、自测；若已配置 **toolbar 主题切换**，可切主题看一遍。  
4. 确认满意后，在 `@gepick/client`（或其他业务包）中 `import` 使用，并保证应用入口或根布局已 **`import '@gepick/ui/styles/globals.css'`**（与「产出形态」一致）。  
5. 在**真实页面**再跑一遍交互，然后提 PR。

## 技术选型

**具体版本以 `package.json`、锁文件、`catalog` 为准**；与实现不一致时以代码为准并更新本文。

| 领域 | 选型 | 说明 |
|------|------|------|
| 框架 | **React** | `react` / `react-dom` 以 **peer** 约版本。 |
| 可访问性原语 | **Radix**（`@radix-ui/react-*`） | 无样式原语，按需子包。 |
| 样式 | **Tailwind CSS** + **@tailwindcss/vite** | token 与 `src/theme`、`@theme`、CSS 变量对齐。 |
| 变体与 class | 可选 **CVA**、**clsx** | 与 Radix `className` 组合。 |
| 开发期 | **Storybook**（`@storybook/react` + Vite） | 与宿主 Vite 链一致。  

安装命令、脚本等**工程细节**以各 `package.json` 的 `scripts` 为准。

## 工程结构

说明 **`packages/ui` 在仓库里怎么放**；与下章「怎么 import」区分。

| 路径 | 职责 |
|------|------|
| `ui/.storybook` | Storybook 配置/预览等，**仅开发期**。 |
| `ui/src/components` | 可发布组件；`*.stories.*` 可与组件同目录（以团队规范为准）。 |
| `ui/src/styles` | 如 `globals.css`、Tailwind 入口，供 Storybook 与宿主入口引用。 |
| `ui/src/hooks` | 可随包导出的 UI 相关 hooks。 |
| `ui/src/theme` | token、主题 Provider、`@theme` 与 CSS 变量等。

## 产出形态

**最终是什么**：和 **Ant Design**、**Element Plus** 一样，就是一个给业务项目 **`import` 用的 React 组件库**——包里有 **组件**、**统一样式入口**、以及**类型**。**不是**另一个独立产品；业务侧也**不用**装 Storybook，Storybook 只在**做库的人**本地/CI 里用。

**调用方在业务里怎么用**（下面 import 路径以本包实际导出为准，名字仅作示例）：

在页面里**当普通第三方 UI 库**用即可，例如整包入口：

```tsx
import { Button, Dialog } from '@gepick/ui'
import '@gepick/ui/styles/globals.css'

export function SaveBar() {
  return (
    <Dialog>
      <Button type="button">取消</Button>
      <Button type="button" variant="primary">保存</Button>
    </Dialog>
  )
}
```

若本库把组件拆成**子路径**（名字随你们定），可能长这样：

```tsx
import { Button } from '@gepick/ui/button'
import '@gepick/ui/styles/globals.css'
```

若本库在 `src/theme` 里提供了**整站主题壳**，一般在应用**最外层**包一次，例如：

```tsx
import { ThemeProvider } from '@gepick/ui/theme'
import '@gepick/ui/styles/globals.css'

export function AppRoot() {
  return (
    <ThemeProvider>
      <业务路由与页面 />
    </ThemeProvider>
  )
}
```

**调用方抓住两点就够**：

- **组件**：从 `@gepick/ui`（或子路径）`import`，在 JSX 里当 React 组件写。  
- **样式**：在应用**入口或根布局**里 **`import` 本库那条全局样式**（通常全站**一次**即可；具体文件名随本库导出）。不写这行，很多 Tailwind 类会**没有 CSS**，页面上会「有组件、没样子」。  

业务工程里照常装好 `react` / `react-dom` 即可；Radix 等由本包依赖带上，**不必**在业务里再开 Storybook。
